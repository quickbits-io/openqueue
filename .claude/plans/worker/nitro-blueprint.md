# Blueprint: Worker replatform onto Nitro (inside PR #37, pre-1.0)

Maintainer decision 2026-07-21 (supersedes "Nitro: not adopted"): match eve's setup —
Nitro as host + build for the worker, h3 apps mounted on it. See plan.md Tech-stack
decisions. Architect run: a8185937dd78aa4b5.

## Design summary

Nitro becomes the **production build + host** for the worker; it does not replace the
worker's HTTP composition. The worker's h3 app assembly (control API, workbench,
health/ready/metrics) is extracted into a listener-free `createWorkerApp`, and gains
two hosts:

1. **Embedded/dev** — `startWorkerApp` keeps its exact signature but swaps `Bun.serve`
   for h3's `serve` (srvx), making `@openqueue/worker` Node-capable. The three e2e
   suites (which boot `startWorkerApp` in-process with `port: 0`) pass unchanged.
2. **Production** — `openqueue build` runs Nitro's builder (`createNitro`/`prepare`/
   `build` from `nitro/builder`, the exact API eve uses against the same pinned
   version) over a **generated boot plugin** + a **single `/**` catch-all handler**
   that delegates every request to the same `createWorkerApp` app. `openqueue start`
   spawns `.output/server/index.mjs`, polls `/health`, and forwards SIGTERM.

Why this over alternatives: registering our routes individually in Nitro's router
would create a second routing surface that can drift from the h3 apps (byte-parity
risk); the catch-all guarantees the artifact's HTTP surface is *the same code* the
e2e suites already prove. Eve's worker-thread HMR dev model is **rejected**: our job
state lives in Redis/Postgres (not an in-process world like eve's), so there is
nothing to preserve across dev reloads, and BullMQ consumers inside disposable worker
threads cannot drain (`worker.terminate()` kills mid-job → stalled-job redelivery →
duplicate processing in dev). The existing restart-on-change subprocess model already
runs the full SIGTERM drain path and stays.

Verified facts this design rests on:
- npm `latest` for `nitro` is **`3.0.260610-beta`** — identical to eve's pin. Pin it
  **exactly** (same policy as h3 `2.0.1-rc.25`). Nitro's own h3 dep is
  `^2.0.1-rc.25`, satisfied by our exact pin → one h3 copy.
- Nitro v3 engines: **`^20.19.0 || >=22.12.0`** — this is the *artifact's* Node
  floor, higher than the repo's 20.11.1.
- Nitro v3 handlers are `(event) => Response` with `event.req` a web Request (eve's
  generated handlers: `export default async ({ req }) => POST(req)`); plugins are
  default-exported `(nitroApp) => {}` functions with
  `nitroApp.hooks.hook('close', …)`; **the node-server preset installs no signal
  handling** (per eve's `sandbox-shutdown-plugin.ts` comment) — we install our own.
- h3 `2.0.1-rc.25` exports `serve(app: H3, options?): Server` via srvx `0.11.22`:
  `port: 0` supported, `server.ready()`, `server.url`,
  `server.close(closeActiveConnections?)`, `bun: { idleTimeout }` passthrough, and
  `gracefulShutdown` (default **true** — must be set `false`; we own signals).

## Decisions (a–i)

**(a) Topology.** `openqueue build` emits a Nitro node-server artifact at
`<config.build.outDir ?? '.output'>` (`server/index.mjs` entry). `openqueue start`
prefers the artifact (spawn + `/health` poll, 60s timeout, SIGTERM forwarding with
**no forced SIGKILL** — jobs may drain for minutes; the platform's grace period owns
the bound; a second signal kills immediately); falls back to today's in-process
source boot when no artifact exists. `openqueue dev` is **unchanged** (subprocess
restart-on-change; we never run consumers in worker threads).

**(b) Config + world boot.** `openqueue build` generates
`.openqueue/generated/boot.mjs`:

```js
// Generated by openqueue build. Do not edit.
import {
  clearRegisteredTasks, clearTaskDiscoveryContext, getRegisteredTasks,
  setTaskDiscoveryContext, validateTaskDefinitions,
} from '@openqueue/core';
import { createNitroWorkerPlugin } from '@openqueue/worker/nitro';

clearRegisteredTasks();
setTaskDiscoveryContext("worker/example.ts");
try { await import("../../worker/example.ts"); } finally { clearTaskDiscoveryContext(); }
// … one block per discovered file, in sortTaskFiles order …
const tasks = validateTaskDefinitions(getRegisteredTasks());
// Config is imported AFTER the task snapshot, dynamically: a static import is
// hoisted before clearRegisteredTasks(), and if the config graph touches task
// files the module cache would leave the registry empty (the exact bug the e2e
// harness documents).
const { default: config } = await import("../../worker.config.ts");
export default createNitroWorkerPlugin({ config, tasks });
```

Rolldown compiles the user's TS config + task graph into the output; env is read at
**boot**, not baked. The plugin (in `@openqueue/worker/nitro`) awaits
`createWorkerApp` — world connect, `redis:`-sugar → `worldBullmq` resolution
(unchanged, still the worker's job), consumer start — **blocking Nitro init, so the
port opens only when ready**. Deliberate deviation from "consumers start after
listen": it matches today's worker and eve; boot gating is covered by
`openqueue start`'s health poll / platform readiness probes. Drain: the plugin
registers `nitroApp.hooks.hook('close', drain)` **and**
`process.once('SIGTERM'|'SIGINT', …)` (node-server preset installs none); drain =
`ready=false` → `runtime.close()` (stop intake → drain in-flight → close world, the
semantics `QueueWorkerRuntime.close` at `packages/core/src/runtime.ts:123-127`
already implements) → `process.exit(143|130)`. Idempotent via a `closed` flag.

**(c) HTTP surface.** One catch-all:
`nitro.options.handlers.push({ route: '/**', handler: '#openqueue/worker-route' })` +
`nitro.options.virtual['#openqueue/worker-route'] = "import { nitroWorkerFetch } from
'@openqueue/worker/nitro';\nexport default (event) => nitroWorkerFetch(event.req);"`
+ `nitro.routing.sync()` (eve's exact mechanism). No Nitro-native routes,
`serverDir: false`, `publicAssets: []`, no scan routes → nothing else can answer.
`/health`, `/ready`, `/metrics`, `/openqueue/v1/*`, `/workbench` and the 404 envelope
are byte-identical because they are literally the same `H3` app. rou3
`/**`-matches-root was already verified in the h3 migration. The three e2e suites +
node-imports smoke run against `startWorkerApp` and pass without edits; the artifact
gets its own smoke.

**(d) Runtime matrix.** `@openqueue/worker` becomes Node-capable: after the swap its
only former Bun API (`Bun.serve`, `packages/worker/src/index.ts:113`) is gone.
Engines: `{ "node": ">=20.11.1", "bun": ">=1.2.0" }` — the *package's* floor is h3's;
the **artifact's** floor is Nitro's `^20.19.0 || >=22.12.0` and is documented as such
(nitro's runtime is bundled inside `.output`, not a worker dep — no engines
conflict). `@openqueue/cli` **stays Bun-only** (`Bun.Glob` discovery, `Bun.spawn`,
env loading remain; `nitro/builder` runs under Bun — rolldown is napi, see risks).
`Bun.build` task compilation **dies** (rolldown replaces it).

**(e) What survives.** `dirs`/`tasks` discovery: fully (the CLI's existing
`discoverTaskFiles`/`scanFiles`/`sortTaskFiles` feed the generated imports).
`.openqueue/build/manifest.mjs`: **removed** — the artifact replaces it; the worker's
`resolveTasks` manifest branch (`packages/worker/src/index.ts:263-274`) is deleted as
orphaned. `config.build.outDir` is re-scoped to the Nitro output dir (default
`.output`); `config.build.external` is appended to `traceDeps`. `examples/basic` dev
flow: unchanged. docker-compose e2e infra: unchanged. Dockerfile template: still
`RUN openqueue build` + `CMD ["openqueue","start","--env-file",".env"]`; deployment
docs gain the `node .output/server/index.mjs` multi-stage recipe. OTel ordering:
unchanged in dev/embedded; in the artifact the boot plugin's import graph loads
`@openqueue/core` before any handler executes. No user-facing `instrumentation.ts`
feature exists — do NOT add one.

**(f) Freeze.** `dev|build|start` names, `--config`, `--env-file` preserved;
`dev-worker` unchanged; `migrations print|status` untouched; `OpenQueueConfig`
schema unchanged (only `build.outDir` default/meaning shifts — MIGRATION item);
`startWorkerApp` options/return unchanged. New public surface, minimal and final:
`@openqueue/worker/nitro` exporting exactly `createNitroWorkerPlugin` +
`nitroWorkerFetch` (the `/nitro` name follows the workbench `/h3` rule).
Consumer-visible notes for MIGRATION.md + upgrading-to-1.0.mdx: build output moved
`.openqueue/build` → `.output` (Nitro server bundle); artifact runs on Node
`^20.19 || >=22.12` or Bun; **direct** `node .output/server/index.mjs` boots default
to Nitro's port 3000 unless `PORT` is set (`openqueue start` keeps the 8090 default
by injecting `PORT`/`NITRO_PORT`).

**(g) Packaging.** `@openqueue/cli` gains `"nitro": "3.0.260610-beta"` (exact).
`@openqueue/worker` gains **no** dependency: `nitro.ts` types `NitroAppLike`
structurally (eve's own pattern). Bun API deaths: `Bun.serve` (worker), `Bun.build` +
`Bun.file` manifest sizing (cli). `check:*` gates: unaffected. CI: worker dist joins
the node-imports smoke; new `artifact-smoke` job builds under **Bun**, boots the
artifact under **Node 22**.

## File-level plan

### packages/worker

**`src/app.ts` (new)** — the moved body of `startWorkerApp` minus listener/signals.
Everything currently in index.ts except `startWorkerApp`/`loadConfig` relocates here
(validateConfig, resolveTasks *without* the manifest branch, `bullmqQueues`,
`createWorkbenchForRuntime`, `catalogJob`, `configDirs` WeakMap):

```ts
export interface CreateWorkerAppOptions {
  cwd?: string;
  tasks?: QueueTaskDiscovery | TaskDefinition[];
}
export interface WorkerAppHandle {
  runtime: QueueWorkerRuntime;
  /** Fully mounted h3 app: /health, /ready, /metrics, /openqueue/v1, workbench. */
  app: H3;
  /** Idempotent: ready=false, then runtime.close() (drain consumers → close world). */
  close(): Promise<void>;
}
export async function createWorkerApp(
  config: OpenQueueConfig,
  options?: CreateWorkerAppOptions,
): Promise<WorkerAppHandle>;
```

**`src/index.ts` (modify)** — public surface unchanged (`startWorkerApp`,
`createWorkbenchForRuntime` re-exported from `./app`, `loadConfig`):

```ts
import { serve } from 'h3';
// inside startWorkerApp:
const handle = await createWorkerApp(config, options);
const port = options.port ?? Number(process.env.PORT ?? 8090);
const server = serve(handle.app, {
  port, silent: true, gracefulShutdown: false, bun: { idleTimeout: 30 },
});
await server.ready();
const bound = Number(new URL(server.url ?? `http://localhost:${port}`).port || port);
// close(): await handle.close(); await server.close(true);  — same order as today
// signals: identical SIGTERM/SIGINT wiring as today (options.signals !== false)
return { runtime: handle.runtime, port: bound, close };
```

`gracefulShutdown: false` is load-bearing — srvx defaults it on and would
double-handle signals.

**`src/nitro.ts` (new)**:

```ts
import type { OpenQueueConfig, TaskDefinition } from '@openqueue/core';
import { createWorkerApp, type WorkerAppHandle } from './app';

interface NitroAppLike {
  hooks: { hook(name: 'close', handler: () => Promise<void>): unknown };
}
export interface NitroWorkerOptions {
  config: OpenQueueConfig;
  tasks: TaskDefinition[];
}
export function createNitroWorkerPlugin(
  options: NitroWorkerOptions,
): (nitroApp: NitroAppLike) => Promise<void>;
export function nitroWorkerFetch(request: Request): Promise<Response>;
```

Module-level `handle: WorkerAppHandle | undefined`; the plugin awaits
`createWorkerApp({ ...options.config }, { tasks: options.tasks })`, assigns, hooks
`close`, installs `process.once` signal handlers (exit 130/143). `nitroWorkerFetch`
returns `handle.app.fetch(request)`; a one-line `503 worker booting` guard covers the
externally-ordered window.

**package.json**: add `"./nitro"` export; engines
`{ "node": ">=20.11.1", "bun": ">=1.2.0" }`; description drops "Bun-native".
**tsup.config.ts**: `entry: { index, nitro: 'src/nitro.ts' }` (optionally
`target: 'node20'` per the WS4 note).

### packages/cli

src/index.ts is 925 lines — split the two reworked commands out; the rest stays put.

**`src/nitro-build.ts` (new)** — `build()`:
1. `loadCliConfig()` → `discoverTaskFiles(config, cwd)` (existing, Bun.Glob) → fail
   on zero files.
2. `loadTasksFromFiles(files, cwd)` (existing) for fail-fast validation + summary
   counts — dupes die *before* the Nitro build.
3. Write `.openqueue/generated/boot.mjs` (template above; relative import specifiers
   from the generated file; labels = `relative(cwd, file)` exactly as the old
   manifest). Keep it **outside** Nitro's `buildDir` (`prepare()` cleans that).
4. Nitro build (all from `nitro/builder`, lazy-imported):

```ts
const nitro = await createNitro({
  _cli: { command: 'build' },
  rootDir: cwd,
  buildDir: resolve(cwd, '.openqueue', 'nitro'),
  output: { dir: resolve(cwd, config.build?.outDir ?? '.output') },
  dev: false,
  serverDir: false,
  scanDirs: [],            // if createNitro rejects [], point at .openqueue/generated (no routes/ inside)
  publicAssets: [],
  features: { websocket: false },
  plugins: [bootPath],
  traceDeps: [...resolvableOpenqueuePackages, ...(config.build?.external ?? [])],
});
nitro.options.handlers.push({ route: '/**', handler: '#openqueue/worker-route' });
nitro.options.virtual['#openqueue/worker-route'] = /* delegate, see (c) */;
nitro.routing.sync();
await prepare(nitro); await copyPublicAssets(nitro); await build(nitro); await nitro.close();
```

`resolvableOpenqueuePackages` = every `@openqueue/*` package resolvable from `cwd`
(probe with `createRequire(join(cwd,'package.json')).resolve('<pkg>/package.json')`,
skip failures). **Load-bearing invariant: ALL `@openqueue/*` packages are
externalized-and-traced, never bundled.** Mixed bundling would create two
`@openqueue/core` instances and split the task-registry/`bindQueueRuntime`
singletons — the failure mode the old manifest's `external: ['@openqueue/*']`
guarded against.

5. Post-build: copy `dist/ui` from the resolved `@openqueue/workbench` package into
   `<out>/server/node_modules/@openqueue/workbench/dist/ui` if absent (idempotent).
   `UI_DIST_PATH` (`packages/workbench/src/ui-dist.ts`) is `import.meta.url`-relative
   so with the package traced-external it resolves inside `.output`; the copy covers
   tracers that only carry reachable JS.
6. Print summary: task/queue/schedule counts + output dir size + path. Delete the old
   manifest writer, `importTasksFromManifest`, and the `Bun.build` call.

**`src/artifact-start.ts` (new)** — `start()`:

```ts
const entry = resolve(cwd, config.build?.outDir ?? '.output', 'server', 'index.mjs');
if (!existsSync(entry)) return startFromSource();   // today's start(), minus manifest preference
const port = Number(process.env.PORT ?? 8090);
const child = Bun.spawn([process.execPath, entry], {
  cwd, stdin/stdout/stderr: 'inherit',
  env: { ...process.env, PORT: String(port), NITRO_PORT: String(port) },
});
// forward SIGTERM/SIGINT → child.kill('SIGTERM'); no timed SIGKILL (drain may take
// minutes; platform grace owns the bound); a second signal → SIGKILL.
// poll GET http://127.0.0.1:${port}/health every 250ms, 60s deadline (eve's constants);
// fail → kill child, exit 1. On 200: log "[openqueue] worker ready on :port".
// await child.exited → process.exit(code)
```

**src/index.ts (modify)**: dispatch build/start to the new modules; update help()
for build; dev, dev-worker, init, add, migrations untouched. **package.json**:
`"nitro": "3.0.260610-beta"` (exact).

## Risks

- **Nitro beta pin**: `3.0.260610-beta` is npm latest and eve-proven; every builder
  API named here is exercised by eve against this exact version. Upgrade = manual
  exact-pin bumps gated by the artifact smoke. No caret ranges on a beta.
- **rolldown under Bun**: napi; least-proven link — CI smoke builds under Bun first.
  Escape hatch: `openqueue build` shells out to node for the builder.
- **Registry/singleton split** (silent 0-tasks artifact): prevented by the
  external-and-trace-all-@openqueue/* invariant + smoke catalog tripwire + generated
  module ordering (clear → import tasks → snapshot → dynamic config).
- **UI_DIST_PATH inside .output**: trace + idempotent copy + smoke assertions.
- **Cold boot**: plugin blocks Nitro init; port opens only when world connected;
  `openqueue start` polls 60s; direct deploys use platform readiness probes.
- **Port default drift**: raw node .output defaults to 3000; `openqueue start`
  injects 8090. MIGRATION + deployment docs call it out.
- **Bun-runs-output**: `openqueue start` under Bun executes the node-server bundle
  via Bun's node:http compat; smoke covers Node authoritative. If Bun-exec regresses,
  fallback = prefer a resolvable node binary (do not build preemptively).
- **API-surface commitments**: `@openqueue/worker/nitro` (2 exports) and the .output
  layout freeze at 1.0 — both deliberately thin; catch-all means future route changes
  never touch the Nitro layer.
