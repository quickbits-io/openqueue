# Blueprint — Phase 3: Worlds

> Produced by the architect agent. Stage A is file-level (implement now); Stages B/C/D are contract-level (short follow-up blueprints before each). Full contract sketches for QueueTransport/capabilities/world composition are in the architect's report; this file preserves the complete plan.

## Design summary

A world composes two independently pluggable axes behind one ownership boundary: **`QueueTransport`** (delivery bus — the seam that does not exist today) and **`QueueStore`** (durable state — **already exists**: `QueueStorage` in `types.ts`, unchanged). The transport interface derives strictly from current BullMQ usage in `enqueue.ts`/`worker.ts`/`schedules.ts`/`cancel.ts` — nothing speculative. Capability flags are part of the transport contract with typed degradation (`UnsupportedCapabilityError`); the workbench `QueueManager`'s raw-BullMQ surface (~40 calls) is deliberately NOT in the contract — it stays a bullmq-scoped dashboard feature via a `BullmqTransport.queue()` escape hatch.

**Package layout: zero new packages until Stage C, then exactly one (`@openqueue/world-postgres`).** transport-bullmq and world-local live inside core as internal modules: core can't shed bullmq/ioredis anyway (frozen public surface), and world-local must run under core's vitest/Node tests (→ in-memory, Node-safe, NO bun:sqlite, no fs). world-postgres earns a package: new pg-driver dep + the load-bearing edge/split-topology artifact needing a bullmq-free graph.

**Selection is value-based**: `world: worldPostgres({...})` — a typed factory, not a string. Eve needs strings for build-time resolution; OpenQueue configs are TS modules imported at boot and bundled by `Bun.build`, where dynamic import-by-string is bundler-hostile. Rejected: string module IDs.

**SDK-facing API does not change in Phase 3.** `task()`, `trigger()`, `QueueClient`, `bindQueueRuntime`, `postgresAdapter`, `@openqueue/client`, and the frozen `/openqueue/v1` wire contract are untouched. Stage A adds ZERO public surface (d.ts byte-stable). Public additions land in Stage C (`QueueConfig.world`, factories, `@openqueue/core/world` subpath) and Stage D (control-plane composition).

## Key contracts (summary — full TS sketches in architect report)

- **`QueueTransport`** (`core/src/transport/types.ts`): `id`, `capabilities: { delay, priority, flows, deduplication, remove }`, `enqueue(queue, TransportJobSpec) → { jobId }`, `enqueueFlow(TransportFlowNode)`, `getJob → TransportJobHandle | undefined` (shaped to satisfy cancel.ts's `CancelableJob` — cancel.ts stays untouched), `listDelayed`, `consume(queue, ConsumeOptions) → TransportConsumer`, `close()`. `ConsumeOptions`: `process(ActiveTransportJob)`, `isFinal(err)` (core passes `isNonRetryable`), `onCompleted`, `onFailed(job, err, { final })` (fires EVERY attempt), `onError`, concurrency/maxStalledCount.
- **Capability enforcement in core, before transport calls** (never silent): enqueue throws for delay/priority when unset; enqueueFlow for flows; runs.cancel for remove; schedule creation for delay. Scheduler timer-loop fallback for delay-less transports NOT built in Phase 3 (all three transports have delay: true).
- **`OpenQueueWorld`** (`core/src/world.ts`, internal until Stage C): `{ specVersion (=1), transport, store: QueueStorage (REQUIRED), migrations?, start?(), close() }`; `WorldFactory = (ctx: { namespace }) => OpenQueueWorld | Promise<…>`; `validateWorld()` structural + exact specVersion (eve's package-manifest peer check skipped — lockstep releases + specVersion suffice).
- **Store composition ownership moves**: today's unconditional `createRedisQueueState(redis, durable)` write-through becomes world-bullmq's private composition; world-postgres = pg store direct; world-local = memory store. Catalog Redis-first fallback moves into world-bullmq. Lifecycle (start/close, self-migration) on the WORLD, not the store.
- **world-postgres binding decisions**: fixed pg schema `openqueue` (v1 non-configurable; custom-schema users keep postgresAdapter), embedded migrations as ordered `{ id, checksum, sql }[]` TS constants generated via `drizzle-kit export --sql` (checksums precomputed — no runtime crypto), ~50-line runner in `start()` (advisory lock → CREATE SCHEMA IF NOT EXISTS → `__openqueue_migrations` bookkeeping → per-step transactions → checksum hard-fail), `migrations: 'manual'` verify-only + `openqueue migrations print|status` CLI, additive-only within a major. transport-postgres: SKIP LOCKED + `claimed_until` visibility timeout + heartbeat; capabilities `{ delay: true, priority: true, flows: false (v1 recommendation), deduplication: true, remove: true }`; postgres.js default driver + `db`-injection for Neon/Hyperdrive (Workers).

## Stage A — file-level (implement now)

## Risks

- **Public-surface commitments land at Stage C** (contract proven against 3 transports first): QueueTransport/capabilities/OpenQueueWorld/WorldFactory/validateWorld, three factories, UnsupportedCapabilityError, QueueConfig.world, the world-postgres package name, PLUS operational surfaces: `openqueue` pg schema, `__openqueue_migrations` table, additive-only DDL history.
- **BullMQ semantics pinned by conformance (do NOT "fix")**: attemptsMade convention (0-based in process(), includes current in callbacks); onFailed per attempt; updateData persists across retries; enqueue drain hook BEFORE transport add; `ttl` is an opaque pass-through smuggled via cast today, enforced by nothing — NOT a capability; dedupe-by-jobId best-effort; delayed = eventual not precise; flow ids can't contain `:` (core-side validation, shared by all transports).
- **Test-infra coupling**: partial `vi.mock('bullmq')` factories → lazy BullMQ construction mandatory; e2e harness reaches into `worker['blockingConnection']` → createWorker keeps returning real Worker[].
- **Workbench queue pages stay bullmq-only** via `BullmqTransport.queue()`; degradation UX decided in Stage C.
- **No migrations for existing users all phase**: postgresAdapter tables untouched (`public` schema); world-postgres creates its own `openqueue` schema; postgresAdapter→worldPostgres switch is a user-owned data migration (document).
- **release-please**: world-postgres joins linked-versions; Stage A/B ship as `feat(core)` internal refactors with explicit "no public API change" notes.
