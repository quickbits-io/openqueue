## The idea

Make OpenQueue work like Vercel's eve: one deployable "worker" app (e.g. `worker.expensicat.com`) that boots the worker, HTTP API, and Workbench under a single umbrella, with pluggable authentication strategies. Application SDKs (`@openqueue/sdk` in user apps) talk to that deployment over HTTP and require **zero** database/Redis connections. Storage happens behind a **world**-style abstraction (as in eve / Workflow DevKit), owned exclusively by the deployed worker app.

## Findings

### Track 4 — Eve client protocol + deployment ✅

How a remote client talks to a deployed eve agent, and the single-app deploy story.

**Client is transport-only — zero storage imports.** `eve/client` (`packages/eve/src/client/`) imports nothing but `fetch`-level primitives; no DB, no queue, no broker. Construction is `new Client({ host, auth?, headers?, redirect?, maxReconnectAttempts? })`. `auth` is a small union: `{ bearer }` | `{ basic }` | `{ vercelOidc }`, with token values as `string | () => string | Promise<string>` resolved **per request** (rotating tokens work across stream reconnects). `host` can be an absolute origin (server-to-server) or a same-origin prefix (browser behind a proxy).

**Wire protocol** — versioned prefix `/eve/v1` (`packages/eve/src/protocol/routes.ts`):
- `GET /health`, `GET /info`
- `POST /session` (create), `POST /session/:id` (continue), `POST /session/:id/cancel`
- `GET /session/:id/stream?startIndex=N` — durable, replayable **NDJSON over long-lived fetch** (no WebSocket/SSE). `startIndex` is an absolute event index; negative = tail-relative (`-1` recovers current state from a bare sessionId). Client auto-reconnects on 404/409/425/5xx from the last consumed index.
- Token-capability callback routes (OAuth, terminal) are unauthenticated-by-token.

**Two-handle model**: `continuationToken` = resume handle (one active at a time, stale rejected); `sessionId` = stream/inspect handle. Client persists only a tiny serializable cursor `{ continuationToken?, sessionId?, streamIndex }`.

**Response ergonomics**: `send()` returns a fetch-like `MessageResponse` — ids available immediately, `AsyncIterable` over events, and `.result()` drains to an aggregated terminal result. React/Vue/Svelte hooks (`useEveAgent`) are a thin store over the same session.

**Deployment**: `eve build` compiles everything into one Nitro artifact serving all routes — API, channels, schedules, callbacks under `/eve/`, workflow dispatch under `/.well-known/workflow/`. On Vercel → Build Output + Vercel Cron; self-hosted → `eve start` with in-process schedule runner. State lives in the workflow **world** (managed on Vercel; local disk world at `.eve/.workflow-data` or pluggable e.g. `@workflow/world-postgres` via `experimental.workflow.world` config). Config via env vars only; auth fail-closed by default (placeholder rejects prod traffic until a real policy is set). Proxy lesson: forward *every* prefix the runtime needs or runs silently stall.

**Implications for OpenQueue** (`@openqueue/client` shape):
1. Client never imports `ioredis`/`bullmq`/drizzle — HTTP only; safe in browsers/edge.
2. `Client({ host, auth })` mirroring eve's options, incl. per-request token resolvers.
3. Two handles → `jobId` (inspect) + optional opaque cursor for flows; cursor stays tiny.
4. Endpoint sketch: `POST /oq/v1/jobs`, `GET /oq/v1/jobs/:id`, `GET /oq/v1/jobs/:id/stream?startIndex=N`, `POST /oq/v1/jobs/:id/cancel`, `GET /oq/v1/health|info`. Version the prefix from day one.
5. NDJSON + index-cursor replay over per-job event log (Redis stream) beats WebSocket/SSE for durability and proxy-friendliness; add propagation-window retries (enqueue-accepted → readable race).
6. Offer low-level event iterator + `await job.result()` aggregate helper.
7. Typed job results: type on client, validate authoritatively on worker.

### Track 1 — Eve boot/umbrella model ✅

**One HTTP host, everything is a route.** Nitro 3 (h3) is eve's sole runtime dependency and sole HTTP host; API, channels, schedules, workflow dispatch, homepage, and health are all handlers on one Nitro instance. One artifact, one domain, one deploy.

- **CLI** (`packages/eve/src/cli/run.ts`): `dev` (default), `build`, `start` — all lazy-import `internal/nitro/host.ts`.
- **Dev**: parent process owns the HTTP listener + local workflow world + file watcher; the compiled app runs in swappable **worker threads** (`DrainedNitroDevServer` + `dev-runner.ts`) — HMR = compile new generation, `replaceWorker`, drain old. Session state survives because it lives in the parent-owned world, not the worker.
- **Prod self-host**: one Nitro Node build (`.output/`), `eve start` spawns a single child process, **polls `/eve/v1/health` until 200** before reporting ready, graceful SIGTERM drain with bounded windows (sandbox 15s < grace 20s < SIGKILL).
- **Vercel**: same authored surface, two build surfaces (`__server.func` app + separate workflow functions); cron via Vercel Cron hitting an **unguessable token path** (`/eve/v1/cron/<token>` — the path is the secret).
- **Route contract**: reserved versioned prefix `/eve/v1/*` + `/.well-known/workflow/v1/flow`; proxies must forward both.
- **Discovery**: filesystem grammar (`src/discover/grammar.ts`) — `agent/{tools,channels,schedules,...}/**`, names derived from paths, charset-validated, collisions fail the build. Zero registration boilerplate.
- **Dev/prod parity**: identical authored surface and route code; only adapters swap (world, sandbox backend, scheduler), branching on `process.env.VERCEL`.
- **Stateless per-request runtime**: every request resolves the shared compiled bundle; durable state lives in the world → horizontal scaling is safe, dashboard reflects all replicas.
- **Bonus**: `withEve` lets eve mount inside an existing host app behind a prefix — a mountable-handler story OpenQueue could mirror (workbench already has `buildWorkbenchApiApp`).

Lessons for OpenQueue: single host + worker loop in one process (packages/worker is already close); reserved versioned prefix; health-gated readiness + bounded graceful drain; filesystem job discovery (`jobs/<name>.ts` → queue+processor); capability-token paths for callbacks/cron instead of shared-secret env matching; dev-only "run schedule now" route; keep the HTTP host decoupled from the durable engine behind a narrow interface.

### Track 5 — OpenQueue coupling map ✅

**Key fact: all connection creation is centralized; `task()` is already connection-free.** Nothing connects until `createQueueClient`/`createQueueWorker` runs. `task()` builds a plain definition and defers `trigger`/`schedules` through `bindQueueRuntime` (`core/src/task.ts:154`) — the indirection for an HTTP runtime **already exists**.

- **The seam**: the `QueueClient` interface (`core/src/runtime.ts:49-61`) — `{ catalog, trigger, schedules, runs, spans?, alerts, close }`. An HTTP implementation of this shape, registered via `bindQueueRuntime`, makes `myTask.trigger()` and `client.runs.retrieve()` work with zero Redis/DB.
- **Connection map**: enqueue → `getQueue` → BullMQ `Queue` on shared ioredis (`enqueue.ts:72-95`), drain fan-out writes run state to Redis cache (`state.ts:368-381`) + Postgres if adapter present (`drizzle.ts:1144-1163`). Runs reads: Postgres-first, Redis-cache fallback (`state.ts:345-395`). Schedules: Redis hashes + Postgres + BullMQ tick jobs (`schedules.ts:79-104`). Catalog-by-id: Redis hget (`catalog.ts:73-85`). Connections physically open in `connection.ts:9-25` (3 ioredis clients for worker), `runtime.ts:291-313`. Postgres is never opened by core — user passes a Drizzle `db` into `postgresAdapter`.
- **State inventory**: Drizzle schema = `catalog`, `schedules`, `schedule_instances`, `runs`, `run_events`, `run_spans`, `alert_channels`, `alert_rules` (`drizzle.ts:164-439`). Redis = namespaced catalog/schedules/runs/alerts hashes; runs cache capped 5000, TTL 7d. BullMQ owns actual job payloads/queue mechanics. **Postgres is authoritative when configured; Redis is a write-through cache.**
- **Worker boot** (`worker/src/index.ts:39`): validate config → resolve tasks (manifest/modules/dirs) → `createQueueWorker` (opens Redis, publishes catalog, one Queue per task queue, BullMQ Workers, schedule sync) → health server → optional workbench mount → `Bun.serve` → SIGTERM/SIGINT drain. The umbrella largely exists.
- **Workbench API**: transport-agnostic route table (`workbench/src/api/handlers.ts:559 buildRouteTable`) consumed by Hono AND a raw fetch handler; the dashboard React app is **already a connection-free HTTP client** of these routes (`ui/lib/api.ts`). Reads go through `QueueManager` (raw BullMQ/Redis); enqueue/dynamic-schedule routes go through the injected core runtime.
- **Route gaps for client mode**: no `POST /schedules` (create) or `PUT /schedules/:id`; `POST /test` returns `TestJobResponse` without `runId`; no catalog read route; run reads are BullMQ-shaped (`RunInfo/JobInfo`), not core `QueueRun`.
- **Hard couplings to break for worlds**: enqueue hard-wired to BullMQ/ioredis (`enqueue.ts:39-95`); state layer takes a concrete `Redis` (`state.ts`); schedules require a BullMQ queue; workbench `QueueManager` is pure Redis/BullMQ; catalog resolution reads Redis. Existing abstractions (`QueueCatalogStore`/`QueueScheduleStore`/`QueueRunStore`/`AlertStore`/`QueueDrain`) cover *durable persistence*, not the Redis transport — the missing piece is a transport-level driver interface.

### Track 2 — Worlds (storage/durability abstraction) ✅

A "world" is the pluggable durability backend of the Workflow SDK (`@workflow/*`). Eve consumes the `World` interface, ships two implementations, and adds a dev-only RPC transport so state survives dev-worker hot reloads.

**The contract** (reconstructed from `development-world-protocol.ts` operation table + `validate-world.ts`):
- Identity: `specVersion` (protocol version, currently 5), `getDeploymentId()`, `processExitTriggersQueueRedelivery`.
- **Queue** (durable delivery bus): `queue(name, message, opts)` to publish; `createQueueHandler(prefix, handler)` / `registerHandler(prefix, httpHandler)` for delivery. Retry semantics via `attempt`/`messageId`/`timeoutSeconds`.
- **Five stored entity families**, each a namespaced sub-API: `runs` (durable executions: get/list/setAttributes), `steps` (checkpoints, replayed on recovery), `events` (append-only log, by id or correlationId — backs history + streaming), `hooks` (park/resume points addressable by capability token — approvals, OAuth callbacks), `streams` (per-run named byte streams; `get` returns a live `ReadableStream`).
- Lifecycle: optional `start()` (begins redelivery) / `close()`.
- Validation: `validateWorkflowWorld` structural check (has `createQueueHandler`, `events`, numeric `specVersion`) + `assertWorkflowWorldCompatibility` (world package's declared `@workflow/core` dep must share eve's bundled major/prerelease line — fail-fast).

**Implementations & selection**: `@workflow/world-local` (filesystem, dev + self-host default, data at `.eve/.workflow-data`), `@workflow/world-vercel` (managed), or any package exporting `createWorld()`. Selected via config (`experimental.workflow.world: "@workflow/world-postgres"`), NOT env vars — default chosen at build time (`process.env.VERCEL ? "vercel" : "local"`). Eve generates a Nitro boot plugin that imports the world module, instantiates, validates, installs as process-global singleton (`setWorld`/`getWorld` via globalThis symbol), and registers queue handlers. World-specific secrets (Postgres URL, etc.) are the world package's concern, read from env.

**Who connects**: production — the deployed app owns the world **in-process** (data ops are method calls; delivery arrives as HTTP POSTs to `/.well-known/workflow/v1/flow`). Dev — the CLI **parent** owns the real local world and serves it over an authenticated RPC route (`/eve/v1/dev/internal/workflow-world`, shared secret, timing-safe compare); the disposable Nitro worker holds an interface-faithful HTTP shim generated from the operation table, with a tagged value codec (Date/Uint8Array/Error survive the boundary) and a raw route for live streams. **The client SDK never touches the world — confirmed by import analysis.** Compute/state split: topology differs dev vs prod, semantics don't.

**OpenQueue adoption sketch** (from the report):
1. `@openqueue/world` — an `OpenQueueWorld` interface: `enqueue`/`registerHandler` (bus) + `jobs`/`attempts`/`events`/`hooks`/`logs` sub-APIs + `specVersion` + `start`/`close`. Attempts ≈ eve steps; logs ≈ eve streams.
2. Two implementations: `@openqueue/world-bullmq` (BullMQ/Redis + Drizzle — prod default; reads `REDIS_URL`/`DATABASE_URL` from env) and `@openqueue/world-local` (SQLite/fs, zero external deps — dev default).
3. Worker app is the sole world owner: boot plugin instantiates + validates + `setStore()`; BullMQ `Worker` callback wires `registerHandler` directly in-process (optional internal HTTP deliver route for serverless topology later).
4. SDK/client: pure HTTP against `/oq/v1/*`; imports nothing from `@openqueue/world*`.
5. Dev parity: CLI parent owns local store (`.openqueue/.job-data`) over authenticated RPC + codec, worker holds shim → job state survives reloads.
Key eve files to copy the pattern from: `internal/workflow/{validate-world,world-compatibility,development-world-{protocol,server,client,codec},local-world-data-directory,queue-namespace}.ts`, `internal/application/compiled-artifacts.ts` (world selection + generated plugin), `internal/nitro/host/configure-nitro-routes.ts`.

### Track 3 — Eve auth strategies ✅

Eve splits auth into two independent systems: **route auth** (inbound, at the channel layer, fails closed) and **connection/tool auth** (outbound, when the agent calls external services).

**Inbound core — the `AuthFn` ordered walk** (`packages/eve/src/public/channels/auth.ts`):
- `AuthFn = (request) => SessionAuthContext | null | Promise<…>` — the single plug-point.
- `routeAuth()` walks an ordered array: first non-null result wins; `null` skips to next; thrown `UnauthenticatedError`/`ForbiddenError` short-circuits 401/403; exhausted list → 401 with `WWW-Authenticate` challenge. **Fails closed, including empty array.**
- Uniform principal output: `{ authenticator, principalId, principalType, issuer?, subject?, attributes }` — carried into runtime as `ctx.session.auth`.
- Shipped strategies (each returns an `AuthFn`): `localDev()`, `vercelOidc()`, `httpBasic()`, `jwtHmac()`, `jwtEcdsa()`, `oidc()`, `none()`, `placeholderAuth()` (401s only in prod — scaffold guardrail so half-configured deploys fail closed).
- Each strategy = pure verifier (`{ok, sessionAuth}`, unit-testable without HTTP) + thin `AuthFn` wrapper. Verifiers in `runtime/governance/auth/*.ts`.
- Eve only **verifies** tokens; it never mints them — bring your own IdP/session. `GET /health` is always public.
- Per-surface policy: each channel factory takes its own `auth` array; dashboard/terminal UI share the eve channel's walk (no separate auth surface).

**Platform webhooks (Slack/Discord/GitHub/Telegram/Twilio/Linear)**: NOT the AuthFn walk — fixed per-platform signature verification over the **raw body, constant-time**, identity derived only from the verified signature. `credentials` picks the secret source (string/env/async provider); `webhookVerifier` escape hatch fully replaces the check (used by Vercel Connect for forwarded webhooks). Schedules run under a synthetic framework principal (`eve:app`). Framework-minted callbacks (OAuth, terminal) auth via capability token in URL path + **SSRF guard** rejecting private/reserved hosts.

**Outbound (connections/tools)**: one abstraction across MCP + OpenAPI — `ConnectionAuthProvider`: non-interactive `{ principalType, getToken() => {token, expiresAt?}, evict? }` or interactive OAuth (`defineInteractiveAuthorization`), or a per-caller resolver `(ctx) => provider`. Tokens resolved/cached per step, never persisted, never shown to the model. `principalType: "user"` binds outbound identity to the inbound principal (schedule-triggered runs fail `principal_required` — correct default).

**Secrets**: no framework store — `process.env` + a user-owned provider interface (multi-tenant pattern: `TenantCredentialProvider`, backend deliberately unprescribed).

**Recommended OpenQueue design** (from the report):
1. Copy the `AuthFn` walk verbatim as `AuthStrategy => Principal | null`, ordered, fail-closed, with a `placeholderAuth`-style prod guard. Pure verifier + wrapper split.
2. Per-surface strategy arrays on the worker config: `enqueueApi: { auth: [apiKey(), hmacSigned(), serviceJwt()] }`, `dashboard: { auth: [appSession(), oidc(), localDev()] }`, per-provider webhook verifiers (constant-time raw-body HMAC + `verifier` override), `health: public`.
3. Stamp the verified `Principal` (incl. `tenantId`) onto the job record at enqueue; enforce ownership ACLs on status/cancel/dashboard reads separately from authentication (authn ≠ ownership).
4. Outbound job-handler auth as `getToken`/`TokenResult` providers, cached per run, never persisted/logged; user-scoped tokens bound to the enqueueing principal.
5. Job-completion webhooks to user URLs: capability token in path + SSRF guard (copy `session-callback.ts`).
6. Drop: Vercel OIDC/Connect specifics, sandbox credential brokering.

## The plan

### Target architecture (end state)

```
┌─ user's app (Next.js, anywhere) ──────────────┐
│ import { task } from '@openqueue/sdk'         │
│ import { createClient } from '@openqueue/client'  ← fetch-only, zero Redis/DB
│ myTask.trigger(input)  → HTTP                 │
└───────────────┬───────────────────────────────┘
                │ POST/GET /openqueue/v1/*  (bearer / api-key / custom AuthStrategy)
┌───────────────▼───────────────────────────────┐
│ worker.expensicat.com — ONE deployed app      │
│ (@openqueue/worker, Bun.serve, one port)      │
│  • /openqueue/v1/*  versioned control API     │
│  • /workbench       dashboard (existing)      │
│  • /health          always public             │
│  • BullMQ Workers   in-process                │
│  • schedule ticks   in-process                │
│  • sole owner of the world (storage)          │
└───────────────┬───────────────────────────────┘
                │ OpenQueueWorld interface
     ┌──────────┴──────────┐
     │ world-bullmq (prod) │  world-local (dev)
     │ Redis + Drizzle/PG  │  zero-dep, on-disk
     └─────────────────────┘
```

Everything eve taught us maps onto seams OpenQueue **already has**: `QueueClient` + `bindQueueRuntime` is the client seam; the transport-agnostic `RouteDef` table is the API seam; `packages/worker` is already the umbrella process; the store interfaces in `core/types.ts` are half a world already.

### Design decisions

1. **Versioned control API at `/openqueue/v1/*`**, mounted by the worker app independently of whether the dashboard is enabled. Response shapes derive from **core types** (`QueueRun`, `EnqueueResult`), not the dashboard's BullMQ-shaped `RunInfo/JobInfo` — dashboard routes stay internal. The prefix is a frozen contract from day one (eve lesson).
2. **`@openqueue/client` is a new package**, fetch-only, safe for Node 18+/Bun/edge. It implements the existing `QueueClient` interface and registers via `bindQueueRuntime`, so `myTask.trigger()` in user code works unchanged with zero connections. `@openqueue/sdk` re-exports it under a subpath (`@openqueue/sdk/client`) so the flagship import stays the one users write. Auth mirrors eve's `ClientAuth`: `{ bearer }` | `{ basic }` | custom header, token values as per-request resolver functions.
3. **Two-handle model**: enqueue returns `{ runId, jobId }`; `runId` is the durable inspect handle (survives BullMQ job retention), `jobId` the transport handle. The client persists at most a tiny cursor.
4. **Auth = eve's `AuthFn` walk, copied faithfully**: `AuthStrategy = (req) => Principal | null`, ordered array, first-match wins, throw → 401/403, exhausted → 401, **fail-closed** (empty array included), `placeholderAuth`-style prod guard. Pure verifier + thin wrapper split for testability. Per-surface arrays: `api.auth`, `workbench.auth`, webhooks per-provider raw-body constant-time verifiers, `health` always public. OpenQueue **verifies, never mints** — bring your own IdP/keys. The verified `Principal` is stamped onto the run at enqueue; ownership ACLs are enforced separately from authentication.
5. **Worlds fold transport + persistence into one ownership boundary.** `OpenQueueWorld` = delivery bus (`enqueue`/`registerHandler`) + stores (`jobs`, `attempts`, `events`, `logs`, `schedules`, `catalog`, `alerts`) + `specVersion` + `start`/`close`. Current BullMQ+ioredis+Drizzle code becomes `@openqueue/world-bullmq` (prod default); `@openqueue/world-local` is a zero-external-dep dev world. Only the worker app instantiates a world; selection is config-driven (`world: '@openqueue/world-bullmq'`) with structural validation + specVersion compatibility (mirror `validate-world.ts`). SDK/client never import world packages.
6. **Sequencing principle: client mode ships before the worlds refactor.** The connection-free SDK doesn't need worlds — the worker keeps using Redis directly; worlds only change who owns storage *internally* and are invisible to SDK users. This de-risks the deep refactor and lets the wire contract inform the world interface.

**Phase 3 — Worlds** *(amended 2026-07-16: two-axis design)*

A world is NOT a monolith — it composes two independently pluggable axes, avoiding the M×N package explosion (kafka+postgres, kafka+mysql, rabbitmq+postgres, …):

- **`QueueTransport`** (delivery bus): enqueue, delay, retry/redelivery, consume, remove-from-queue. Implementations: `transport-bullmq` (Redis — default, wraps today's code), `transport-kafka`, `transport-rabbitmq`, `transport-postgres` (SKIP LOCKED polling).
- **`QueueStore`** (durable state: runs/attempts/events/schedules/catalog/alerts): **this axis already exists in core** — the `QueueStorage`/store interfaces (`types.ts:244-519`) with `postgresAdapter` as the first implementation. Add `store-mysql` (Drizzle already supports MySQL) and `store-sqlite` (local/dev).
- **World packages are thin presets** binding the axes: `world-bullmq` = transport-bullmq (+ optional store), `world-postgres` = transport-postgres + store-postgres (single-dependency deploy — one database powers jobs AND state), `world-kafka` = transport-kafka + store of choice, `world-local` = in-proc/sqlite both axes (dev default). Usage: `createWorker({ world: worldPostgres({ url }) })`.
- **Capability flags are part of the transport contract** — this is load-bearing, not decoration. Kafka has no native delayed delivery; flows/parent-child and priorities are BullMQ-specific; repeatable ticks need a scheduler loop when the transport lacks delay. Transports declare `capabilities: { delay, priority, flows, deduplication, … }`; core features degrade with **typed errors** (never silent no-ops) when a capability is absent; the conformance suite parameterizes over capabilities; schedules fall back to a worker-internal timer loop for delay-less transports.
- Extract the `QueueTransport` seam from `connection.ts`/`queue.ts`/`enqueue.ts`/BullMQ `Worker` wiring; workbench `QueueManager` raw-BullMQ reads get scoped as bullmq-transport-only features initially (dashboard degrades gracefully on other transports).
- **Zero-migration requirement** *(added 2026-07-16, from eve)*: eve ships **no ORM** — its only runtime dep is nitro; storage tech is entirely the world package's concern. Users never run migrations because (a) the dev/self-host default world is *filesystem* storage (`.eve/.workflow-data` — no DB to migrate), (b) Vercel's world is a managed service, and (c) `@workflow/world-postgres` reads its connection string from env and owns its schema lifecycle — the docs never mention a migration step ("There's nothing to configure"). OpenQueue worlds must match this: **configure `world-postgres` and it just works.** Concretely: embedded versioned migrations inside the world package, self-applied on `world.start()` (`CREATE SCHEMA IF NOT EXISTS` + internal version table), guarded by a Postgres advisory lock so N booting replicas don't race, with an opt-out (`migrations: 'manual'` + a CLI command that prints the SQL) for ops teams that review DDL. No drizzle-kit in user projects, ever — whether we keep Drizzle internally as a query builder is an implementation detail. Today's `postgresAdapter({ db })` (bring-your-own-Drizzle-db, user-owned migrations) stays as the power-user escape hatch but stops being the default path.
- **Migration mechanics** *(decided 2026-07-16)*: migrations are **maintainer-generated, user-invisible** (pg-boss/graphile-worker pattern). Generation: we run `drizzle-kit generate` against `defineQueueSchema()` at dev time, review, and check the SQL into the world package as ordered TS string constants (bundler-safe; no `.sql` assets); a CI parity check (`drizzle-kit generate` → empty diff) prevents schema/DDL drift. Runtime: our own ~50-line runner in `world.start()` — advisory lock (replica boot race), `CREATE SCHEMA IF NOT EXISTS`, `__migrations` bookkeeping table with checksums (hard-fail on history edits), each step in a transaction (Postgres transactional DDL). Additive-only within a major for N-1 rolling-deploy compatibility. `migrations: 'manual'` = verify-only + `openqueue migrations print|status` CLI. ORM decision: **Drizzle stays as an internal implementation detail** of the store packages (already in the tree; dialect coverage pg/mysql/sqlite matches the store axis); it never leaks past the `QueueStore` interface, so a later swap (Kysely's programmatic `MigrationProvider` being the best alternative) is non-breaking. Prisma/TypeORM rejected (codegen/runtime weight in an infra package).
- **Deployment topologies** *(added 2026-07-16 — "run the API on Workers")*: worlds must enable a **split topology**, not just the single umbrella. (a) Today: one worker app owns everything. (b) Phase 3 target: **control plane on serverless/edge** — a `ControlRuntime` constructed over a world directly (no BullMQ consumers booted), so `buildControlApp` (Hono, already Workers-native) deploys to Cloudflare Workers/Vercel edge; execution plane stays persistent. Requirement this imposes: store/transport *enqueue+read* paths must work on request-scoped runtimes — `world-postgres` needs a Workers-compatible driver (postgres.js/Neon/Hyperdrive); `transport-bullmq` enqueue can't (ioredis TCP + Lua) — postgres-first sequencing is therefore load-bearing. (c) Far horizon: `world-cloudflare` — CF Queues **push consumers** are a real serverless delivery transport (the eve HTTP-callback analog) + Workers-safe store = fully serverless execution within Workers CPU limits (IO-bound jobs yes, heavy compute no). Note: Hono covers every host here — this is a connections problem, not an HTTP-framework problem; the Nitro decision stands.
- Sequencing inside the phase: extract interfaces + conformance suite → wrap bullmq as `transport-bullmq` (zero behavior change) → `world-local` → `world-postgres` (validates the whole design incl. self-migration + the serverless control-plane topology; needs the Phase 1.5 compose stack) → kafka/rabbitmq/mysql later, demand-driven.
- Verify: full core suite green on `world-bullmq` AND `world-local`; conformance suite green per transport with correct capability skips; workbench functional on both.

**1.0 checklist item** *(added 2026-07-17)*: consider extracting `worldBullmq` into `@openqueue/world-bullmq` at the first major — core sheds `bullmq`/`ioredis` entirely (becoming edge-clean itself), the Bull dependency becomes opt-in like Postgres is. Blocked pre-1.0 by core's frozen Bull-flavored surface (`createQueue`/`createConnection`/`createWorker`/`redis:` config). The Stage B/C blueprints deliberately kept this door open; conformance suite + world contract make the move mechanical when the time comes. Also on the 1.0 list: Node 18 floor drop for core/sdk (unblocks jose v6), h3 stable-release unpin.

**Phase 4 — Umbrella polish + dev parity**
- **`withOpenqueue` (embedded mode)** *(added 2026-07-16, mirroring eve's `withEve`)*: a helper that boots the engine in-process and returns mountable handler(s) so an existing **long-running** server (Bun/Node/Hono monolith, self-hosted Next with a custom server) hosts the control API + dashboard under a prefix without a second deploy. Thin wrapper — the pieces already exist (`buildControlApp`, `buildWorkbenchApp`, the Next adapter). Explicitly NOT for serverless hosts in v1: BullMQ needs a persistent consumer; the serverless-embedded story unlocks in Phase 3 via a transport with HTTP-callback delivery (eve's `/.well-known/workflow/v1/flow` analog). Document the reserved-prefix rule and auth wiring.
- **Framework-neutral mounting** *(added 2026-07-17 — "not all users have Hono")*: the interop surface is the web-standard fetch handler, which everything modern consumes directly (Workers/Bun/Deno/Next/Hono) and the route layer already supports below Hono (`RouteDef` table + `createFetchHandler`, Hono-free — the Tauri-sidecar path). The gap is legacy Node frameworks (Express/Koa/Fastify): ship a `toNodeHandler()` bridge (~30 lines or `@hono/node-server`/`@whatwg-node/server`) + per-framework mounting recipes in docs. **Decision: h3/Nitro migration rejected for this** — an h3 app needs the identical Node bridge for Express embedding; swapping frameworks changes nothing for users (Hono is an internal detail they never touch) and is pure churn. Fetch-handler-first is the policy.
- NDJSON run-event streaming (`GET /openqueue/v1/runs/:id/stream?startIndex=N`, index-cursor replay, negative tail reads) + client `stream()`/`await run.result()` aggregation, propagation-window retries.
- Health-gated readiness in CLI/deploy docs; bounded graceful-drain audit; capability-token callback URLs + SSRF guard for job-completion webhooks.
- Dev parity: CLI-parent-owned local world so job state survives dev reloads (eve's dev split); dev-only schedule run-now already exists (`POST /schedules/:id/run`).

### Tech-stack decisions (2026-07-16, amended 2026-07-17)

Revisit exactly when Phase 3 makes serverless execution real; even then it's a deploy-preset question, not a framework one. - **SUPERSEDED — Nitro: adopted, pre-1.0 (maintainer decision, 2026-07-21).** The maintainer wants the worker to match eve's full setup: Nitro as host + build ("i want a similar setup as eve"), landing inside PR #37 before the 1.0 merge. The h3 apps are already Nitro-native (h3 is Nitro's handler layer), so this is a host/build replatform of `@openqueue/worker` + `@openqueue/cli` (`Bun.serve` → Nitro host; CLI `Bun.build` pipeline → Nitro build), not an HTTP-surface change. ### Risks & open questions

- **API freeze**: `/openqueue/v1` shapes are forever once published — review the wire types before Phase 1 merges.
- **Client runtime target**: recommended Node 18+/Bun/edge (fetch-only). Browser usage is possible but auth guidance (no secrets in browsers) needed — dashboard already covers human use.
- **`world-local` runtime**: if built on `bun:sqlite` it's Bun-only (fine — only the Bun-only worker instantiates worlds); a Node-portable fallback (JSON/fs) is possible if self-host-on-Node ever matters.

- **Versioning**: new packages (`client`, `world`, `world-bullmq`, `world-local`) join the lockstep changesets release train.
- **Open**: exact package name for the client (`@openqueue/client` vs sdk-subpath-only); whether `spans`/`alerts` APIs make the v1 wire contract or stay dashboard-internal.
